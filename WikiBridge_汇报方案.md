# WikiBridge 课程项目汇报方案 / 讲稿

> 适用场景：向课程老师进行现场演示汇报，建议总时长 8–10 分钟。

---

## 一、项目一句话定位（开场 30 秒）

**WikiBridge = “零代码个人知识库 + 一键公网发布 + AI 问答消费”三件套。**

它能把本地散落的 PDF、网页、文档自动编译成一本带有 `[[双向链接]]` 的 Markdown Wiki；再把这本 Wiki 通过 frp 隧道发布到公网；最后让读者用浏览器或 OpenCode AI 对话界面进行检索与问答。

**给老师留下的印象**：这不是一个单纯的前端页面，而是一个覆盖 **采集 → 加工 → 存储 → 检索 → 可视化 → 发布 → 消费** 全链路的完整系统。

---

## 二、整体架构图（讲解 1 分钟）

```
┌─────────────┐   编译    ┌─────────────┐   代理    ┌─────────────┐
│   发布端    │ ────────→ │  llm_wiki   │ ←─────── │  opencode   │
│ (BearFRP/   │  LLM生成  │  Wiki 服务器 │ KB模式桥接 │  AI 消费端  │
│  Desktop)   │           │  :19828     │           │  :4096      │
└─────────────┘           └──────┬──────┘           └──────┬──────┘
                                 │                         │
                                 └───────────┬─────────────┘
                                             ↓
                                      ┌─────────────┐
                                      │    nginx    │
                                      │    :80      │
                                      └──────┬──────┘
                                             ↓
                                      ┌─────────────┐
                                      │   bearfrp   │
                                      │  公网隧道   │
                                      │ :8000/7000  │
                                      └─────────────┘
```

**技术栈速览**：

- `llm_wiki`：Rust + Tauri v2 + React 19 + Vite + LanceDB 向量检索 + sigma.js 图谱
- `opencode`：Bun + Effect + SolidJS 的 AI 对话中台，在本项目里运行在 KB 沙箱模式
- `bearfrp`：Python + FastAPI + frp v0.58.1 多租户隧道平台
- `desktop`：Tauri 桌面壳，把上面三个二进制文件打包成侧载进程
- 部署：`docker compose up --build -d` 一键启动

## 三、完整演示脚本（8–10 分钟）

### 场景 0：Docker 一键启动（30 秒）

**操作**：在终端输入：

```bash
cp .env.example .env
docker compose up --build -d
docker compose logs bearfrp-wikibridge-frpc
```

**讲稿**：
> “老师，首先我们看部署。整个系统用 Docker Compose 一键拉起，包含 Wiki 服务器、AI 消费端、nginx 网关、frp 隧道控制面和自动注册的 frpc 侧载。日志里会吐出一个公网 URL，待会我们直接用它访问。”

**印象点**：从代码仓库到可公网访问的产品，只需要一条命令。

---

### 场景 1：进入发布端，创建知识库项目（1 分钟）

**操作**：

1. 打开桌面端 `发布端` 或浏览器访问 `http://localhost`。
2. 点击 **创建项目**，选择本地文件夹，输入项目名（例如 `课程论文资料库`）。
3. 选择模板（如 Research / Reading）。

**讲稿**：
> “这是发布端。一个知识库就是一个本地文件夹项目。创建时可以选择场景模板，系统会写入 `purpose.md` 和 `schema.md`，相当于给 LLM 设定‘这本书要写什么、怎么写’。”

**印象点**：用 Markdown 元数据约束 LLM 输出，保证生成的 Wiki 风格一致。

---

### 场景 2：拖拽导入素材，看实时构建队列（1.5 分钟）

**操作**：

1. 在 `Sources` 页面点击 **Add 文件 / Add 文件夹**。
2. 拖入 1 个 PDF、1 个 TXT/MD、1 张图片（如果支持）。
3. 观察右侧 **Activity 活动面板** 的进度条、状态、失败原因。

**讲稿**：
> “我导入三类素材：文本型 PDF、纯文本、网页剪藏。系统会先在本地提取 PDF 文本（不依赖 OCR），然后进入两阶段链式构建：第一步 LLM 分析这篇材料‘在讲什么’，第二步根据分析结果生成或更新 Wiki 页面。Activity 面板实时显示队列、进度、错误，失败项可以重试或忽略。”

**印象点**：

- 两阶段链式思维（分析 → 生成）
- SHA256 增量缓存，重复文件不会重复构建
- 本地 PDF 解析，保护隐私

---

### 场景 3：浏览生成的 Wiki 与双向链接（1 分钟）

**操作**：

1. 切换到 **Wiki** 视图。
2. 打开一个自动生成的概念页。
3. 点击页面里的 `[[另一个概念]]` 跳转到对应页面。
4. 展示 YAML frontmatter（类型、标签、来源）。

**讲稿**：
> “构建完成后，我们得到一本 Markdown Wiki。每一页都有 YAML frontmatter，标记页面类型——是实体、概念、来源还是综合。点击 `[[双向链接]]` 可以在知识点之间跳转，这就把线性文档变成了网状知识。”

**印象点**：LLM 不仅提取摘要，还建立知识之间的关联。

---

### 场景 4：知识图谱可视化（1.5 分钟，核心亮点）

**操作**：

1. 切换到 **Graph** 视图。
2. 等待 sigma.js 力导向图渲染。
3. 切换着色模式：按 **类型** → 按 **社区（Louvain）**。
4. 点击一个节点，高亮相邻节点。
5. 打开右侧 **Insights** 面板，展示：
   - Surprising connections（意外连接）
   - Isolated pages（孤立页面）
   - Bridge nodes（桥梁节点）
   - Knowledge gaps（知识缺口）

**讲稿**：
> “这是我最喜欢演示的功能。系统根据四种信号计算页面相关性：直接链接、来源重叠、Adamic-Adar 相似度和类型亲和度。然后用 Louvain 算法自动发现社区。老师您看，这里不仅是一张图，右侧还会给出‘知识洞察’——比如这两个社区之间有一个桥接节点，或者某块知识是孤立的，可以直接点击触发深度研究去补全。”

**印象点**：

- 不是静态图，是带分析建议的知识图谱
- 社区检测 + 桥接节点识别
- 图谱缺陷可直接驱动 AI 研究

---

### 场景 5：搜索：文本 + 图片多模态检索（1 分钟）

**操作**：

1. 切换到 **Search** 视图。
2. 输入查询词（例如某个概念名）。
3. 展示文本结果摘要高亮。
4. 展示图片结果网格，点击图片进入灯箱。
5. 点击“跳转到源文档”。

**讲稿**：
> “搜索不是简单字符串匹配。系统做分词关键词搜索，如果配置了 Embedding 模型，还会做 LanceDB 向量语义搜索。更关键的是，它支持图片——PDF 里被提取出来的图片会显示缩略图，点开灯箱，还能一键回到原始 PDF 出处。”

**印象点**：多模态检索 + 可溯源。

---

### 场景 6：AI 问答与引用（1 分钟）

**操作**：

1. 切换到 **Chat** 视图。
2. 新建对话，输入问题，例如：“总结 `overview.md` 的核心观点。”
3. 观察流式输出、`<think>` 思考块折叠、引用卡片。
4. 点击引用卡片跳回 Wiki 原文。

**讲稿**：
> “消费者可以像用 ChatGPT 一样提问，但答案不是幻觉。模型会调用 `llm_wiki_search` 和 `llm_wiki_read_file` 工具先查知识库，再基于检索结果回答，每个论断都附带引用卡片，点卡片就能回到原文。”

**印象点**：RAG + 工具调用 + 可验证的回答。

---

### 场景 7：Deep Research 深度研究（1 分钟）

**操作**：

1. 打开右侧 **Research** 面板。
2. 输入一个开放问题，例如：“Transformer 架构在知识图谱构建中的最新应用。”
3. 观察联网搜索、摘要、来源列表。
4. 保存结果到 Wiki。

**讲稿**：
> “当知识库有缺口时，可以一键启动深度研究。系统会去网上搜索（支持 Tavily/SerpApi/SearXNG），把搜索结果交给 LLM 综合，并保存成新的 Wiki 页面，实现知识库的自我生长。”

**印象点**：从“消费知识库”升级到“自动扩展知识库”。

---

### 场景 8：一键公网发布（1 分钟）

**操作**：

1. 切到桌面端 **发布端 → 访问连接**。
2. 选择要发布的项目，点击 **发布**。
3. 系统通过 BearFRP 创建 TCP 代理，生成公网 URL。
4. 复制 URL，在浏览器新标签页打开。

**讲稿**：
> “最后看发布。我们自己搭了一个 frp 管理平台 BearFRP，支持多用户、流量配额、端口分配、token 轮换。点击发布后，后台自动注册 frpc 隧道，给我们一个公网地址。老师您看，我现在用浏览器访问这个地址，不需要任何域名或服务器配置。”

**印象点**：

- 自带内网穿透平台
- 公网 URL 一键生成
- 多租户、流量控制、安全配置

---

### 场景 9：消费端：OpenCode 对话访问（1 分钟）

**操作**：

1. 切到桌面端 **消费端**。
2. 配置 DeepSeek / OpenAI API Key。
3. 粘贴刚才的公网 Wiki URL 作为远程知识库。
4. 提问，观察模型调用 Wiki 工具回答。

**讲稿**：
> “公网发布后，别人怎么消费？我们提供两种入口：普通浏览器和 OpenCode AI 对话。在消费端里，用户填写 API Key、粘贴 Wiki 地址，就可以像聊天一样查询。OpenCode 运行在 KB 沙箱模式，只能读写自己的知识库，不能执行 shell，保证了安全。”

**印象点**：发布者与消费者分离，访问受控。

---

### 收尾总结（30 秒）

**讲稿**：
> “WikiBridge 的完整链路是：本地素材 → LLM 编译成带双向链接的 Wiki → 知识图谱与多模态检索 → AI 问答与深度研究 → frp 公网发布 → 安全的消费端访问。它把一个‘资料整理’的小需求，做成了覆盖采集、加工、检索、可视化、发布、消费六个环节的完整系统。”

## 六、可能的老师提问与回答准备

**Q1：LLM 生成的内容质量怎么保证？**
> 我们通过 `purpose.md` 和 `schema.md` 约束输出格式；生成后进入 Review 队列，人工可接受/拒绝/修改；还有 Lint 扫描死链、孤立页、重复页。

**Q2：和 Notion/Obsidian 有什么区别？**
> Obsidian 是编辑器，WikiBridge 是“AI 编译器”——自动把原始材料变成结构化 Wiki，并自带发布和 AI 消费端。

**Q3：多人协作怎么做？**
> 当前版本是单用户本地项目；公网发布后多人只读消费。后续可扩展为 Git 同步或多用户项目锁。

**Q4：为什么不用现成的向量库？**
> 我们用 LanceDB 本地向量库，零依赖、零配置、保护隐私，且支持增量 upsert 和删除。

**Q5：frp 安全性如何？**
> BearFRP 有用户注册、token 轮换、frps 插件回调校验、流量配额、管理员面板；OpenCode KB 模式限制文件访问范围并禁用 shell。
