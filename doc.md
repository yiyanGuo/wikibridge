# 零代码内网穿透本地知识库平台：项目简介文档

## 一、项目选题与背景意义

### 1. 痛点场景：知识管理的“孤岛”与“高门槛”

在当前的数字化学习和办公场景中，个人和小团队通常会持续积累大量资料，包括文档、网页剪藏、聊天记录、课堂笔记、灵感片段等。但这些资料往往分散在不同软件和文件夹中，难以长期沉淀，也难以在后续使用时快速找到知识之间的联系。

我们观察到三个主要问题：

1. **知识零散难归档**：资料分散在 Word、PDF、网页、聊天记录和个人笔记中，传统文件管理只能保存文件，难以建立概念之间的深层联系。
2. **AI 知识库部署成本高**：现有 RAG 或私有知识库方案通常需要配置 Python 环境、向量数据库、模型服务甚至 GPU，对教师、文字工作者、中小团队等非技术用户并不友好。
3. **本地知识库共享困难**：即使用户在本地搭建了知识库，也很难安全、低成本地发布到公网，供其他设备、同学、团队成员或外部访问者使用。

因此，本项目希望解决的问题不是单纯“做一个知识库”，而是让普通用户能够低门槛完成从资料整理、知识结构化到公网访问的完整闭环。

### 2. 选题价值：本地编译与轻量分发的闭环

本项目拟开发一个“零代码、可本地编译、可无感发布”的个性化知识库平台。与传统实时动态 RAG 不同，我们将知识库生成过程类比为“软件编译”：

- 用户先将原始资料放入本地素材目录。
- 用户主动触发编译，系统调用大模型 Agent 对素材进行阅读、提炼、去重和结构化。
- 编译结果输出为带有双向链接的 Markdown Wiki 页面。
- 本地 Wiki 通过轻量 Web 服务渲染为静态站点。
- 平台借助 frp 隧道将本地站点映射到公网。
- 外部访问者通过 URL 浏览知识图谱，并可使用 Chatbox 进行问答。

这一方案形成了清晰的价值闭环：

- **价值输入**：用户把日常积累的文档、笔记、网页片段放入本地素材箱。
- **价值重塑**：LLM-Wiki 编译流程将零散素材整理成结构化知识图谱。
- **价值分发**：frp 无感穿透把本地静态知识站点发布到公网。
- **价值消费**：访问者无需安装软件，通过浏览器即可阅读和提问。

核心优势在于：用户数据和主要计算过程留在本地，云端只负责账号、鉴权、端口和流量调度，不保存用户知识内容，从而兼顾低成本、隐私性和易用性。

## 二、核心方案与工作流

本项目的核心机制是用户主动触发的 **Add / Compile / Lint** 工作流。

### 1. Add：素材箱管理

用户端软件提供本地素材目录，例如 `/raw`。用户可以将 Word、PDF、网页摘录、文本片段等资料放入该目录，也可以通过后续扩展的剪藏入口持续加入素材。

Add 阶段的目标是降低资料收集成本。用户不需要立即整理，也不需要理解知识库内部结构，只需要把未处理资料先放入素材箱。

### 2. Compile：本地知识编译

当用户点击“开始编译”后，系统读取 `/raw` 中的新素材，调用大模型 Agent 作为“知识主编”，完成以下任务：

- 阅读和理解新素材。
- 提取核心实体、概念和主题。
- 将重复或相近内容合并。
- 生成结构化 Markdown 页面。
- 在相关概念之间建立 `[[wiki-links]]` 双向链接。
- 将结果写入 `/wiki` 目录。

Compile 是本项目区别于传统知识库方案的关键。它不是简单把资料塞入向量数据库，而是把零散资料重构为可阅读、可链接、可长期维护的 Wiki 文本。

### 3. Lint：图谱检查与维护

当用户点击“检查优化”后，系统扫描 `/wiki` 目录，对现有知识图谱进行维护：

- 检查死链接和孤立页面。
- 合并冗余页面。
- 修复不一致的标题和链接。
- 提示结构过散或主题重复的问题。

Lint 的意义在于让知识库可以长期生长，而不是随着素材增加变得混乱。

### 4. Publish：无感公网发布

用户端内置轻量 Web Server，将 `/wiki` 目录渲染为静态网页，并在本地端口提供服务。与此同时，后台运行改造后的 frpc，根据用户登录状态向服务器申请公网映射。

服务器通过 frps 和管理 API 分配子域名或端口，最终生成一个可访问的公网 URL。用户可以直接分享该 URL，外部访问者即可浏览知识库。

## 三、三端架构

项目整体分为服务器、用户端、用户的用户三个端。

### 1. 服务器：平台控制与隧道调度

服务器不存储用户的知识库内容，主要负责平台层面的控制能力：

- 用户注册、登录和 Token 鉴权。
- 动态分配公网子域名或端口。
- 管理 frps 服务和隧道连接状态。
- 统计基础带宽、连接时长和访问状态。
- 实现基础限速和防滥用策略。

服务器承担的是“控制面”职责，而不是“业务数据面”职责。

### 2. 用户端：知识编译与本地托管

用户端是知识拥有者使用的软件，是本项目的核心操作入口。它需要屏蔽技术细节，让普通用户通过图形化界面完成：

- 平台账号登录。
- 素材箱管理。
- Add / Compile / Lint 主动触发。
- 编译进度和日志查看。
- 本地 Markdown 静态站托管。
- frpc 后台连接和公网 URL 展示。

用户端同时运行知识库生成逻辑、本地 Web 服务和网络隧道客户端，但这些细节对用户应尽量无感。

### 3. 用户的用户：知识消费端

知识消费端不需要安装任何软件，只需要通过浏览器访问用户分享的公网 URL。主要能力包括：

- 静态 Wiki 页面浏览。
- 通过双向链接在概念之间跳转。
- 使用 Chatbox 进行基于知识库上下文的问答。
- 查看回答引用来源，方便回到原始 Wiki 页面验证。

消费端的核心目标是快速、轻量、低成本。普通浏览行为应主要由静态页面完成，只有用户主动提问时才产生 AI 计算开销。

## 四、需求分析

### 1. 功能性需求

#### A. 用户端 GUI 软件

1. **账号与网络控制**
   - 支持用户通过平台账号登录。
   - 登录后自动向服务器申请公网映射。
   - 展示当前知识库的公网访问 URL。

2. **素材箱管理**
   - 提供 `/raw` 原始素材目录入口。
   - 支持用户持续加入未处理资料。
   - 后续可扩展浏览器剪藏或拖拽导入。

3. **手动编译控制**
   - 提供明确的“开始编译”按钮。
   - 编译时调用 LLM 处理新增素材。
   - 在 `/wiki` 目录生成或合并 Markdown 页面。
   - 界面展示实时进度、日志和错误提示。

4. **图谱维护**
   - 提供“检查优化”按钮。
   - 扫描现有 Wiki 页面。
   - 修复死链接、冗余页面和结构问题。

5. **本地服务托管**
   - 内置轻量 Web Server。
   - 自动将 `/wiki` 目录渲染为静态网页。
   - 在本地端口提供访问服务。

#### B. 服务器管理面板

1. **用户认证系统**
   - 支持用户注册、登录和 Token 管理。
   - 校验 frpc 客户端的身份。

2. **动态隧道管理**
   - 根据用户 Token 分配子域名或端口。
   - 管理 frps 与 frpc 的连接状态。
   - 支持公网访问入口的创建、查看和回收。

3. **资源与流量控制**
   - 统计隧道带宽、连接时长和访问状态。
   - 提供基础限速和防滥用能力。
   - 与用户知识内容保持隔离。

#### C. 知识库消费端

1. **静态图谱浏览**
   - 支持快速加载 Wiki 页面。
   - 支持概念页面之间的双向链接跳转。
   - 支持目录、搜索或基础导航。

2. **交互式 Chatbox**
   - 支持基于当前知识库上下文提问。
   - 支持流式输出回答。
   - 支持标注引用来源。

### 2. 非功能性需求

1. **易用性**
   - 用户端软件应尽量做到双击即用。
   - 不要求普通用户手动安装 Python、Docker 或配置命令行环境。
   - 部署、编译、发布流程尽量图形化。

2. **安全性与隐私**
   - 云端服务器不存储用户笔记、文档、Wiki 页面或向量数据。
   - 用户业务数据主要保留在本地。
   - 公网访问入口需要可控，支持鉴权和回收。

3. **性能与成本**
   - 静态 Wiki 浏览应尽量做到零 Token 消耗。
   - 只有 Compile、Lint 或 Chatbox 主动问答时才产生 AI 计算开销。
   - 云端只承担轻量调度和网络转发，避免高额 GPU 成本。

4. **可靠性**
   - 编译失败、网络断开、端口占用时应提供清晰提示。
   - frp 隧道应支持状态检测和断线重连。
   - 本地服务和公网映射状态应在界面中可见。

5. **可维护性**
   - 知识库生成模块与 frp 管理模块保持解耦。
   - 服务器控制面与用户本地业务数据分离。
   - 后续可扩展不同知识库渲染器、不同客户端和不同发布策略。

## 五、项目开发计划

本阶段按 3-4 周课程设计原型周期规划，目标是完成可演示的端到端闭环，而不是一次性实现完整商业产品。

| 阶段 | 核心目标 | 关键输出物 |
| --- | --- | --- |
| 第 1 周：需求确认与架构设计 | 明确用户角色、三端边界、Add / Compile / Lint 工作流和发布链路。 | 需求说明、系统架构图、核心流程图、原型界面草图。 |
| 第 2 周：本地编译与静态站原型 | 实现素材读取、Markdown Wiki 生成和本地静态网页渲染。 | `/raw` 到 `/wiki` 的最小编译脚本、本地 Web Server、基础 Wiki 页面。 |
| 第 3 周：隧道发布与账号联调 | 接入 frp 隧道、用户鉴权和公网 URL 分配，打通外部访问链路。 | frps / frpc 联调结果、动态端口或子域名分配、可访问公网演示地址。 |
| 第 4 周：体验优化与汇报准备 | 完善日志、错误提示、Lint 检查、演示数据和阶段汇报材料。 | 可演示端到端原型、测试记录、项目简介文档、阶段汇报 PPT。 |

阶段优先级如下：

1. **先打通主链路**：从本地素材到 Wiki 页面，再到公网 URL 访问。
2. **再完善可控性**：账号、Token、端口、连接状态和访问入口可管理。
3. **最后优化体验**：减少用户配置，完善日志、错误提示和演示效果。

## 六、阶段目标与总结

本阶段项目目标是验证“LLM-Wiki 本地编译 + frp 无感穿透”的可行性。我们希望证明：普通用户不写代码，也可以把自己的零散资料整理成结构化知识库，并通过一个公网 URL 分享给他人使用。

阶段一的汇报重点包括：

- 选题具有明确现实需求：知识分散、AI 知识库门槛高、本地共享困难。
- 技术方案具有清晰闭环：Add 素材积累、Compile 本地编译、Lint 图谱维护、Publish 公网发布。
- 系统架构边界明确：服务器负责控制面，用户端负责知识生成和本地托管，消费端负责浏览和问答。
- 开发计划聚焦原型验证：在 3-4 周内优先完成端到端可演示闭环。

一句话概括：本项目要做的是一个面向低技术门槛用户的本地知识库平台，让用户能够零代码完成知识整理、Wiki 编译和公网发布。
